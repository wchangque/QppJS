## P1 — 热路径性能问题

- **每次变量访问（kGetVar/kSetVar）都通过链式 shared_ptr 解引用到 Cell 再读值**（`src/runtime/environment.cpp:53-74`，`include/qppjs/runtime/environment.h:18-19`）
  - 问题：`Cell` 用 `std::shared_ptr<Cell>` 包装，每次读写变量都要：`unordered_map` 查找 → `Binding` → `shared_ptr<Cell>` 解引用 → `Cell::value`。`shared_ptr` 本身没有原子计数开销（非多线程），但多一层堆间接是实质性的 cache miss 来源。更关键的是，`Environment::get` 返回 `EvalResult::ok(b->cell->value)`，这里拷贝了一次 `Value`（含 `std::variant`，若持有 `std::string` 或 `ObjectPtr` 则触发 `std::string` 拷贝或 `shared_ptr` 引用计数递增）。
  - 影响：kGetVar 是 VM 主循环中最高频的指令之一。每次循环变量读取（`i < n`、`i++` 等）都走这条路径，当前规模下已经是实质性开销。
  - 建议：移除 `Cell` 间接层，`Binding` 直接内联持有 `Value`；`get` 返回 `const Value*` 或 `Value&` 避免拷贝，由调用方决定是否复制。

- **`kGetVar` 结果拷贝到操作数栈**（`src/vm/vm.cpp:441`）
  - 问题：`stack.push_back(result.value())` 中 `result.value()` 返回 `const Value&`，但 `push_back` 接受 `const Value&` 会拷贝。若 Value 是 String 类型，触发 `std::string` 深拷贝；若是 Object，触发 `shared_ptr` 原子引用计数递增。VM 主循环内每次读取变量都付出这个代价。
  - 影响：对于纯数值循环（`i < n`），Value 是 `double` 类型，`variant` 拷贝代价相对较小；但对字符串密集场景或对象访问场景，每次 GetVar 都是一次堆分配级别的操作。
  - 建议：在 `Cell`/`Binding` 改为直接持有 Value 后，可以在 `get` 路径上返回 `const Value&`，再由 VM 侧决定 `push_back(std::move(...))` 或浅拷贝。

- **kCall/kCallMethod/kNewCall：每次调用都堆分配 `std::vector<Value> args`**（`src/vm/vm.cpp:636-641`，`673-680`，`714-720`）
  - 问题：三个调用指令都在循环体内执行 `std::vector<Value> args; args.reserve(argc);`，即每次函数调用都触发一次堆分配。`std::reverse` 也是额外的 O(n) 操作。
  - 影响：函数调用是 JS 引擎最频繁的操作之一（递归、回调、方法链）。每次调用都付出 `malloc` 代价。
  - 建议：使用栈上小缓冲区（SBO，如 `SmallVector<Value, 8>`）；或直接从操作数栈的尾部切片传递参数，避免单独分配；或将参数以正序压栈并在 push_call_frame 时直接按索引访问。

- **kPushScope 每次创建新 `shared_ptr<Environment>`**（`src/vm/vm.cpp:491`）
  - 问题：每个块语句（`{}`）执行时调用 `std::make_shared<Environment>(env)`，触发堆分配。循环体内的块作用域（如 `for` 循环体）每次迭代都分配一个新 Environment。
  - 影响：对于含块作用域的循环，N 次迭代 = N 次 `make_shared` + N 次 `unordered_map` 构造 + N 次析构。这是当前最容易触发的热路径问题。
  - 建议：对不含 `let`/`const` 的块跳过 PushScope（编译器优化）；或使用 Environment 对象池；或将 scope 表示为栈帧内的 offset 而非独立对象。

- **`unordered_map<std::string, Binding>` 作为 Environment 的绑定存储**（`include/qppjs/runtime/environment.h:26`）
  - 问题：`std::unordered_map` 的每次 `find` 需要对 `std::string` 进行哈希计算和 `operator==` 比较，且存在哈希冲突的桶链表遍历。对于函数体内参数量通常 ≤ 8 的情况，线性搜索 `std::vector<std::pair<std::string_view, Binding>>` 可能更快（更好的缓存局部性，无哈希开销）。
  - 影响：每次 kGetVar/kSetVar 都经过 `unordered_map::find`，在参数/局部变量少的函数中，哈希开销占比显著。
  - 建议：对小规模绑定（≤ 16 个）使用线性扫描的 flat map，仅在超出阈值时升级到 `unordered_map`。

- **`to_string_val` 在 kAdd（字符串拼接）路径上使用 `std::ostringstream`**（`src/vm/vm.cpp:79-85`）
  - 问题：每次数字转字符串都构造 `std::ostringstream`，这是一个重量级对象（内部含 `locale`、`rdbuf` 等）。在字符串拼接密集的代码中（如 `"" + i`），每次循环迭代都付出这个代价。
  - 影响：字符串拼接循环中，每次数字到字符串的转换都触发 `ostringstream` 构造/析构。
  - 建议：使用 `std::to_string` 或 `snprintf` 到栈上缓冲区，避免 `ostringstream`。

- **`Value::kind()` 在 `to_boolean`/`to_string_val` 等辅助函数中重复调用 `std::holds_alternative`**（`src/runtime/value.cpp:20-37`）
  - 问题：`kind()` 通过 6 次 `std::holds_alternative` 顺序探测来返回枚举值，而 `std::variant` 内部已经存储了类型索引（通常是 `index()`）。这是 O(6) 的线性探测，而直接用 `storage_.index()` 是 O(1)。
  - 影响：`to_boolean`、`to_string_val`、`strict_eq` 在 VM 主循环的比较/控制流指令中被频繁调用。
  - 建议：将 `kind()` 改为基于 `storage_.index()` 的 O(1) 查表实现，或直接用 `std::visit` 替代先查 kind 再 switch 的两步模式。

- **`kLoadNumber`/`kLoadString` 拷贝常量池中的 Value**（`src/vm/vm.cpp:411-419`）
  - 问题：`stack.push_back(bc->constants[idx])` 拷贝 `Value`。若常量是字符串，每次执行该指令都触发 `std::string` 深拷贝。在循环内的字符串常量（如 `"hello"` 被反复加载）每次迭代都分配一次。
  - 影响：字符串常量在循环中使用时，每次迭代一次 `std::string` 分配。
  - 建议：常量池中的字符串改用引用计数字符串（如 `std::shared_ptr<std::string>` 或自定义 RC string），使 LoadString 只递增引用计数而非深拷贝。

- **`dynamic_pointer_cast<JSObject>` 在 kGetProp/kSetProp/kGetElem/kSetElem 中频繁使用**（`src/vm/vm.cpp:534`、`567`、`586`、`606`）
  - 问题：`std::dynamic_pointer_cast` 依赖 RTTI 的 `dynamic_cast`，每次执行需要遍历类型层次。属性访问是 JS 引擎最高频操作之一。
  - 影响：每次属性读写都付出 RTTI 代价。
  - 建议：利用已有的 `object_kind()` 虚函数做 tagged dispatch，用 `static_cast` 替代 `dynamic_pointer_cast`；或将 `object_kind()` 的结果缓存在 Object 基类的非虚字段中，完全消除虚函数调用。

## P2 — 设计层技术债务

- **`Value` 使用 `std::variant` 而非 NaN-boxing**（`include/qppjs/runtime/value.h:56`）
  - 问题：`std::variant<Undefined, Null, bool, double, std::string, ObjectPtr>` 的 sizeof 由最大成员决定（`std::string` 通常 32 字节，加上 variant 的 discriminant），导致每个 Value 约 40 字节。NaN-boxing 可将 Value 压缩到 8 字节（一个 `double`），操作数栈的内存占用减少 5x，cache 命中率大幅提升。
  - 影响：当前阶段操作数栈每个元素 40 字节，100 个元素的栈占 4KB；NaN-boxing 后仅 800 字节，完全在 L1 cache 内。这是 JS 引擎最核心的数据结构选型问题，影响所有路径。
  - 建议：长期目标是迁移到 NaN-boxing（QuickJS 的核心设计）；短期可先将 `std::string` 替换为引用计数字符串以减少 Value 的拷贝代价。

- **`ObjectPtr = std::shared_ptr<Object>` 导致所有对象操作都有原子引用计数开销**（`include/qppjs/runtime/value.h:19`）
  - 问题：每次 Value 拷贝（操作数栈 push、函数参数传递、属性读取返回）若持有 Object，都触发 `shared_ptr` 的原子 `fetch_add`/`fetch_sub`。单线程场景下原子操作比普通整数操作慢约 5-10 倍。
  - 影响：所有对象密集型代码路径（OOP 风格的 JS）。
  - 建议：迁移到 GC 或基于 arena 的对象管理，彻底消除引用计数；或至少改为非原子引用计数（单线程保证下）。

- **`Environment` 的 `outer_` 字段是 `shared_ptr<Environment>`，形成共享所有权链**（`include/qppjs/runtime/environment.h:59`）
  - 问题：闭包捕获外层 Environment 时，整条 scope chain 都被 `shared_ptr` 持有，生命周期管理复杂且每次 scope 创建/销毁都有原子计数操作。在深嵌套函数或大量闭包的场景下，这是显著的内存和性能压力。
  - 影响：闭包密集型代码（回调、高阶函数）。
  - 建议：将 Environment 生命周期与 CallFrame 绑定（非闭包场景），仅在需要闭包捕获时才进行堆分配和引用计数。

- **`add_name` 在编译期做 O(n) 线性去重扫描**（`src/vm/compiler.cpp:82-91`）
  - 问题：每次 `add_name` 调用都线性扫描 `current_->names` 向量。对于有 k 个不同名字、总引用次数为 m 的函数，总代价是 O(m*k)。编译期开销，不影响运行时，但大型文件编译会变慢。
  - 影响：编译阶段。当前代码规模下不显著，但随着文件规模增大会成为瓶颈。
  - 建议：编译期维护一个 `unordered_map<string_view, uint16_t>` 作为名字池的反向索引，将 `add_name` 降为 O(1)。

- **`parse_number_text` 中从 `string_view` 构造临时 `std::string` 再传给 `std::stod`**（`src/frontend/parser.cpp:169-173`）
  - 问题：每次解析十进制数字字面量都从 `string_view` 构造一个临时 `std::string` 再传给 `std::stod`。解析阶段开销，不在运行时热路径上。
  - 影响：仅解析阶段，影响有限。
  - 建议：使用 `std::from_chars`（C++17）直接从 `string_view` 的 `data()/size()` 解析，避免临时字符串分配。

## 结论

- 发现问题：P0 0 条，P1 9 条，P2 5 条

- P1 优先级排序（改动成本 vs 收益）：

  1. **kPushScope 每次循环迭代分配 Environment**（vm.cpp:491）— 改动成本低（编译器端跳过无 let/const 的块），收益高（直接消除循环体内的 make_shared）。
  2. **Cell 间接层 + kGetVar 值拷贝**（environment.h:18-19，environment.cpp:53-63，vm.cpp:441）— 移除 `shared_ptr<Cell>`，`Binding` 直接持有 Value，改动涉及 environment.h/cpp，收益极高。
  3. **dynamic_pointer_cast 在属性访问热路径**（vm.cpp:534, 567, 586, 606）— 改动成本低（改用 object_kind() + static_cast），收益高。
  4. **kCall/kCallMethod/kNewCall 每次调用分配 args vector**（vm.cpp:636, 673, 714）— 改动成本中等，收益高（消除每次函数调用的 malloc）。
  5. **Value::kind() 的 O(6) 线性探测**（value.cpp:20-37）— 改动成本极低（一行改为 storage_.index() 查表），收益中等。
  6. **to_string_val 中的 ostringstream**（vm.cpp:79-85）— 改动成本低，收益在字符串密集场景下明显。
  7. **LoadString 拷贝字符串常量**（vm.cpp:411-419）— 需要引入 RC string 类型，改动成本较高。
  8. **unordered_map 作为 Environment 绑定存储**（environment.h:26）— 改动成本中等，收益在小函数场景下明显。
  9. **kGetVar 结果路径上的 Value 拷贝**（vm.cpp:441）— 依赖第 2 条的 Cell 改造，同步处理即可。
