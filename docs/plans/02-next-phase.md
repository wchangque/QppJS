# QppJS 下一阶段计划

本文件展开“下一阶段”的可执行内容，是从当前状态进入下一轮开发的直接行动卡。

## 1. 下一阶段

- 下一阶段：Phase 0：项目骨架与实验环境
- 对应路线图：`docs/plans/00-roadmap.md`
- 当前事实源：`docs/plans/01-current-status.md`

## 2. 阶段目标

建立一套适合长期学习与持续迭代的最小工程基础，让后续 lexer / parser / runtime 工作有稳定落点。

## 3. 进入前提

当前已具备：
- 明确的项目目标
- 参考仓库与研究方向
- agent team 协作方式
- 路线图与状态文档体系

因此可以正式进入工程搭建阶段。

## 4. 本阶段任务分解

### 0.1 建立最小目录结构
目标：
- 创建 `src/`、`tests/`、`include/` 基础布局
- 先只建立空骨架，不急于填充所有模块

建议结果：
- `src/main/`
- `src/runtime/`
- `src/lexer/`
- `src/parser/`
- `include/qppjs/`
- `tests/unit/`
- `tests/integration/`

### 0.2 建立最小构建链路
目标：
- 选定初版构建方式
- 让空工程可以成功构建

建议：
- 初版使用 CMake
- 先让一个最小可执行文件编译通过

### 0.6 建立测试基线
目标：
- 让测试命令尽早可跑
- 为后续 parser/runtime 演进打基础

建议：
- 先接入一个轻量测试框架
- 至少能运行一个 smoke test

### 0.3 建立最小 CLI
目标：
- 提供第一个可以运行的程序入口

建议：
- 先支持 `qppjs "1+2"` 这类最小命令行输入
- 第一版可以只打印收到的源码或参数

### 0.4 设计错误处理基础结构
目标：
- 统一错误表示，避免后续 parser/runtime 各写一套

建议：
- 先区分语法错误、运行时错误、内部错误
- 第一版只要结构清晰，不追求复杂封装

### 0.5 设计第一版 Value
目标：
- 给后续 interpreter/object/function 提供基础值表示

建议：
- 支持 `undefined`、`null`、`bool`、`number`、`string`、`object`
- 初版偏简单实现，不做优化版表示

### 0.7 建立调试输出入口
目标：
- 为早期 lexer/parser 调试预留接口

建议：
- 预留 `dump_tokens` / `dump_ast` 风格的输出入口

## 5. 建议执行顺序

建议严格按以下顺序推进：
1. 0.1 建立最小目录结构
2. 0.2 建立最小构建链路
3. 0.6 建立测试基线
4. 0.3 建立最小 CLI
5. 0.4 设计错误处理基础结构
6. 0.5 设计第一版 Value
7. 0.7 建立调试输出入口

原因：
- 先让工程能建、能测，再做具体结构设计
- 这样后面每加一小步都可验证

## 6. 验证方式

本阶段完成后，应至少能验证：
- 项目可成功构建
- 最小可执行程序可运行
- 测试命令可执行
- 基础错误类型与 Value 结构可编译通过

## 7. 暂不处理内容

本阶段不进入：
- 词法分析实现
- AST 节点实现
- 解释执行
- 对象模型
- VM
- GC

## 8. 退出条件

满足以下条件即可进入 Phase 1：
- `src/`、`tests/`、`include/` 目录骨架已建立
- 构建系统可用
- 测试框架可用
- 最小 CLI 可运行
- 有第一版 `Error` 与 `Value` 基础结构
